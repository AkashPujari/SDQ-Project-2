Car Service Record Management System

Introduction


Purpose:

The goal of this test plan is to set a baseline for future testing efforts for this system. This plan will serve as the foundation for the minimum testing to be completed, as well as provide direction on the overall testing strategy, including the types of testing to be performed, various phases of testing, exit criteria, and so on. 

This document should be an add-on to the system requirements and the architectural changes, and it should be the baseline for all the testing-related tasks.


System overview

This is a web application system that keeps track of auto maintenance, which will be helpful when buying or selling a certain car to learn more about it and its condition. This technology will assist in keeping track of data regarding used or accident-damaged vehicles.


Scope:

This test plan's scope is confined to integration and Validation Testsing. We have provided a detailed explanation of the requirements, design features, and test plan. We have a distinct phase in our test plans for each unique test method that includes a brief description of the purpose of building all test case skeletons for the test cases. The traceability matrix serves as the foundation for all of our test cases.


System Architecture

Based on the system's requirements, we identified five main components. The database, REST API (authentication module, DAO Module, and Request Handler Module), and The five core components are the five core components. Each of these components belongs to one of three tiers in a multi-tier design. The data tier is in charge of data storage and retrieval. The database module is included in this tier. All business logic, including permission and access control, is handled by the application tier. This tier is made up entirely of the REST API, which is made up of the Authentication module, the DAO module, and the Request Handler module. The presentation tier serves as the system's external interface. The web application is part of this tier. In the presentation tier, the web application talks to the REST API modules in the application tier, and the REST API talks to the database module in the data tier.

Test Plan & Integration


Our system makes use of a three-layered N-Tier architecture. In this case, lower levels are acting as a support system for layers above them. Because of the reliance this architecture creates, components in lower layers must be constructed before those in higher layers. In light of these circumstances, we will divide the system build process into three separate phases, each of which corresponds to a different layer of the system architecture. 

Separating testing into distinct phases will improve the process as a whole and ensure that it grows and evolves in conjunction with the system. It will also open the door to automation, which can eventually result in continuous integration, in which every change is incorporated instantly via an automated pipeline that also handles building, packaging, and testing. That way, any problems with integration may be found and fixed before the code is even written.

Following this methodology, we will begin our test process at the lowest layer—the database or data access layer—move up to the application programming interface layer, then go on to the presentation layer, also known as the graphical user interface layer. We will be able to firm the foundations for higher levels by testing the lower ones first since we are following the route of dependency between these layers. The three build phases that we will test are as follows:

  •  Web application Layer: In the Presentation layer, we'll concentrate on the GUI's usability and accessibility. In order to make sure the system meets the non-functional criteria, we would additionally test its performance.

  •  Data base layer: We will concentrate on testing data access objects in the data access layer to make sure they are operating as intended. We would also concentrate on evaluating the database's performance and security features.

  •  Backend Layer: Testing efforts will be concentrated on functional and security testing to make sure APIs are secure and operating in accordance with specifications. More attention must be paid to security since APIs will be made accessible to the outside world through a programming interface.


Database Phase

Database layer is not covered by unit testing so it is important to include this because the bugs that were not caught during unit testing will be discovered here. Since it is the layer underneath all others, the data access layer's health has a direct bearing on the health of the other layers. After a lengthy integration process with many levels, it may be difficult to find any persisting flaws. In this phase we have two different components that we need to test. The following four points will serve as the basis for all of our test cases:

Data Mapping: Make sure there is a one-to-one correspondence between the fields in the data access layer and their corresponding fields in the database.

Data Integrity: There should be no inconsistencies in the data between different database transactions.

ACID Properties: Atomicity, Consistency, Isolation, and Durability are its abbreviated meanings. These four conditions must be true for every database transaction.

Backend Phase

Test Procedure

The validation of the functional requirements depends on this layer because it contains all of the business logic. The majority of the functional criteria may actually be tested in part programmatically using the APIs. Each test case run on the Backend should at the very least be compared to the specification for the following:
  • Correct HTTP Response Headers
  • Correct HTTP Status Code
  • Correct HTTP Response Payload
During this build, we will test focus on following testing methodologies as part of our Backend testing strategy:
  • Functional Testing: To ensure all functionality has been implemented as intended
  • Security Testing: To ensure authentication and authorization has been properly implemented
  • Performance Testing: To ensure we meet the performance SLAs as specified
  • Load Testing: To ensure we continue to function within SLAs when under load
