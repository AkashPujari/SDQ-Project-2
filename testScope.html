Car Service Record Management System

Test Scope


Test Items

The components that we will be considering for the scope of testing are:
  • Web applications must be tested for GUI testing, which will cover a variety of GUI elements like usability, functionality, accessibility, and more.
  • Customers and apps from third parties can communicate directly with the system via the Rest API. APIs should go through functional and security testing.
  • A hidden element that is not visible to the user is the database. Security testing should also be carried out to make sure that data leaks and sql injection are not possible.

Functional Requirements

REQ# Requirement Description In Scope? Testing Goal/Description
FR-1 The system should be able to allow general users, admins, car insurance companies, workshops, and car manufacturer companies to login and sign up. Yes The user should be able to create an account on the web application.
FR-2 General users should be able to update their personal information. Yes The user should be able to update their records using the correct login credentials.
FR-3 General users should be able to request car information using the car registration number and owner details. Yes Users will be able to request car information.
FR-4 General users should be able to accept or reject any request made to their profile, either for car information or personal information, from general users or car insurance companies. Yes Users will have the ability to accept or reject that will be made to their profile.
FR-5 The system should allow workshops to create, update, or delete maintenance records created by them. Yes The users from workshops should be able to create, update and maintain the records.
FR-6 The system should be able to provide car manufacturers with all the car service records related to their brand and model. Yes The system will be able to provide the manufacturers with the car service records that are related to their model.
FR-7 The system should be able to generate a PDF of the report so that it can be viewed offline. No User should be able to download the pdf report so they can view it offline.
FR-8 Admins should have the ability to register, authorize, or revoke the access of any kind of user. Yes All the admins will have the ability to register, authorize and revoke the access to any user.
FR-9 The system should be able to notify users about the upcoming service date of their cars. No External notifaications are not in our scope of testing
FR-10 The system should allow insurance companies to request a general user for all their car and personal information. Yes The insurance company will be able to request personal information from a user.


Non-Functional Requirements

REQ# Requirement Description In Scope? Testing Goal/Description
NFR-1 The system must be scalable in order to support an expanding user base. Yes The program should undergo load testing to make sure no errors or unexpected results are encountered.
NFR-2 Multiple users looking for information about the same car should be supported by the system. Yes The system should support multiple users searching information about the same car. This can be done by volume testing.
NFR-3 Different browsers should be able to access the website. Yes The web application should be able to run on different browsers.
NFR-4 To enable quick access, the system should store the data in the cloud. No This non-functional requirement is a design principle, hence testing is not permitted.
NFR-5 To provide a quick website load time, numerous servers spread across various locations are required. No This non-functional requirement is a design principle, hence testing is not permitted.
NFR-6 To ensure that no data is lost, there should be a data backup. No This is an operational aspect and will be out of scope for testing.
NFR-7 The appropriate actions should be taken to guarantee the security of data servers. Yes To ensure that no unauthorized connection to the database can be made, penetration testing should be done.
NFR-8 Each page of a website should load in under two seconds. Yes The program should undergo load testing to make sure no errors or unexpected results are encountered. Page load times should be tracked using a standardized third-party application performance monitoring solution, such as Dynatrace or AppDynamics.
NFR-9 The website should be accessible 99.9% of the time. Yes The program should undergo load testing to make sure no errors or unexpected results are encountered. Application availability metrics should be gathered using an agreed-upon third-party tool for monitoring application performance, such as Dynatrace or AppDynamics.


Exit Criteria

Exit Criteria specify when enough testing has been accomplished and when testing may be recognized as complete. We shall apply the following criteria:
  • All planned testing have been completed.
  • The coverage requirements of the code should be satisfied.
  • There should be no outstanding critical or high severity faults.
  • All high-risk regions have been thoroughly evaluated.
